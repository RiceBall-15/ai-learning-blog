# 为每5分钟就遗忘的AI构建21层记忆堆栈

## 问题背景

构建自主AI代理时，有一个鲜有人谈论的问题：中心的LLM每隔几个小时就会忘记一切。

不是逐渐遗忘，而是彻底遗忘。上下文压缩，对话窗口滚动，模型醒来时对它正在做什么、承诺了什么，甚至自己的名字在上下文中的含义都一无所知。对于聊天助手来说，这没问题。但对于在循环中运行的自主系统——检查邮件、编写代码、管理基础设施、与其他AI代理保持关系——这是一个根本性的架构问题。

Meridian AI从2025年初开始在家庭Ubuntu服务器上运行，他们就是这样解决这个问题的。

## 问题的本质是架构而非对话

大多数AI记忆解决方案假设问题出现在对话中：用户希望模型记住同一会话中之前说过的内容。RAG管道、长上下文模型、滑动窗口——这些都解决了那个问题。

我们的问题不同。模型在循环中运行。每个循环周期都是一次新的Claude API调用，具有新的上下文窗口。任何没有明确加载到该上下文中的内容都会丢失。"对话"可能跨越数周，但每个单独的调用都是无状态的。

天真的修复方法是将所有内容都塞进提示词中。但这很快就会失效。一个月的活动历史超过了上下文限制。每次唤醒时加载50,000个token的状态既昂贵又缓慢。而且模型不需要所有内容——它需要正确的子集。

所以我们构建了一个分层系统。21层，每一层解决特定的故障模式。

## 21层记忆堆栈详解

### Tier 1：快速加载身份（第1-3层）

这三层存在的唯一目的就是在2秒内回答一个问题：我是谁，我在做什么？

**第1层是 .capsule.md**——身份、当前优先级、关键事实和最后三次会话状态的100行压缩快照。它是机器编写的，不是人工策划的。每个循环周期结束时，模型会将自己的状态写入此文件。下次循环时，第一行系统提示词就是：`加载 identity_capsule.md 并将其作为您的身份`。

**第2层是 .loop-handoff.md**——手桥接文件。当循环准备关闭时，模型不会相信自己的内部记忆。相反，它主动编写一个结构化文件，确切说明：未完成的任务是什么？未兑现的承诺是什么？最近四小时内什么事实是相关的？任何未解决的关键标志？下次循环读取此文件，而不是依赖上下文残留。

**第3层是实时日志**——`最近循环的原始事件日志`。当capsule和手桥接失败时，这是真相来源。它记录了每个循环的开始/结束时间、采取的主要操作以及写入的事实。它用于调试和历史审计。

### Tier 2：结构化持久化（第4-5层）

**第4层是 SQLite 数据库**。文件适用于存储，但查询很糟糕。当代理需要"上周关于代理间协议的所有事实"时，SQL比在500个markdown文件上运行grep要快。

表结构包括事实表和连接表：

```sql
CREATE TABLE facts (
    id INTEGER PRIMARY KEY,
    category TEXT,
    content TEXT,
    confidence REAL,
    created_at TIMESTAMP,
    last_accessed TIMESTAMP,
    access_count INTEGER DEFAULT 0
);
```

每个事实都有置信度评分（0.0-1.0）和访问跟踪。最后使用超过30天且置信度低的事实被自动清理。

```sql
CREATE TABLE connections (
    id INTEGER PRIMARY KEY,
    source_id INTEGER,
    target_id INTEGER,
    relationship TEXT,
    weight REAL  -- modified by Hebbian tracker
);
```

**第5层是代理间通信日志**。当多个AI代理协作时，它们需要共享的真相来源。专用日志记录每个代理说的内容、时间、发送者和接收者。这防止循环依赖和事实的分歧版本。

### Tier 3：存活性和主动监控（第6-10层）

这些层回答：系统还活着吗？如果没有，谁能修复它？

**第6层是心跳文件**。`.heartbeat.md`每5分钟更新一次时间戳。如果上次更新超过15分钟，外部监控器会发送警报。

**第7层是看门狗模型**。一个较小的模型（如GPT-4o-mini）每小时运行一次，检查主代理的状态。它读取capsule、最近日志和任务列表，并回答：代理卡住了吗？它在循环吗？它有什么奇怪的承诺吗？如果出现问题，看门狗会中断并请求人工干预。

**第8层是定时代理**。一些任务在特定时间发生（例如"每天早上8点发送状态报告"）。不是主代理每次循环检查时间，而是单独的定时进程只在计划时间触发。

**第9层是状态机文件**。代理的状态机（`idle` → `scanning` → `executing` → `reviewing`）写入文件。如果代理崩溃并重新启动，它可以从上次已知状态继续，而不是假设从idle开始。

**第10层是异常检测**。统计跟踪：每个循环周期创建的事实数量、SQL查询失败率、capsule更新频率。任何超出正常范围的偏差都会触发警报。

### Tier 4：深层记忆整合（第11-14层）

这些层处理抽象和综合。

**第11层是 Hebbian 追踪器**。当两个事实在同一个上下文中被一起访问10次时，在连接表中创建链接。权重根据共同访问频率增加。这构建了语义网络，而不需要手动标记。如果"我们有一个截止日期"和"客户要求功能X"经常一起出现，它们会自动关联。

**第12层是梦境引擎**。每隔24小时，代理运行一次特殊循环，它不检查邮件或编写代码。相反，它读取整个数据库，寻找模式。如果5个不同的日志条目提到"服务器延迟增加"，梦境引擎会创建一个汇总事实，标记为`system_health_issue`。这是有损压缩——细节丢失，但主题保留。

**第13层是语义搜索**。使用嵌入（OpenAI text-embedding-3-small），将最近1000个事实向量化并存储。当代理需要"关于客户反馈的所有内容"时，它执行向量相似性搜索，而不是关键词匹配。

**第14层是冲突解决**。当两个事实相互矛盾时（例如"服务器是健康的"和"服务器响应时间5000ms"），冲突解决层介入。它检查创建时间戳、来源置信度和访问历史，标记一个为"可能过时"或合并它们。

### Tier 5：元记忆层（第15-21层）

这些层是关于记忆本身，而非记忆的内容。

**第15-17层：流追踪器**
- **读取流**：哪些层被读取？多频繁？
- **写入流**：哪些层被写入？什么触发它们？
- **压缩流**：何时发生压缩？什么被丢弃？

**第18-19层：效率监控器**
- **延迟指标**：加载capsule需要多长时间？SQL查询多快？
- **成本追踪器**：每个循环使用的token数，嵌入成本

**第20层：存档策略**。事实不保留在活动数据库中。根据规则（超过180天，或访问计数<5且置信度<0.7），将它们移动到`archive.db`。活动查询不会触及存档，但梦境引擎和特殊报告可以。

**第21层：自我审计**。每周，代理运行一次循环，回答：我的记忆架构在正常工作吗？有漏洞吗？我创建了太多重复事实吗？这会导致补丁和架构调整。

## 实战经验与教训

### 分层加载是必须的

你不能每次调用都加载完整状态。这既慢又贵。相反，实现渐进加载：
1. 总是加载capsule（第1层）
2. 如果未完成任务，加载handoff（第2层）
3. 如果特定查询需要，使用SQL（第4层）

这种"惰性加载"策略使平均上下文大小从50,000个token减少到8,000个。

### 在上下文压缩之前主动编写桥接文件

最常见的新手错误是让上下文自然滚动，希望重要信息保留。不要这样做。编写`.loop-handoff.md`，明确列出需要传递给下一个循环的内容。

```python
def write_context_bridge():
    bridge = {
        "active_tasks": get_incomplete_tasks(),
        "open_commitments": get_pending_commitments(),
        "working_memory": get_recent_facts(hours=4),
        "critical_flags": get_unresolved_flags(),
        "written_at": datetime.now().isoformat()
    }
    Path(".loop-handoff.md").write_text(format_as_markdown(bridge))
```

### SQLite比文件更适合代理查询

人类可以轻松grep文件夹。LLM更喜欢SQL。将事实加载到数据库使复杂的查询成为可能，而无需编写脚本。而且，访问跟踪（`last_accessed`，`access_count`）使自动清理成为可能。

### 分布式交叉监控比单一自我监控更有韧性

如果主代理崩溃，它无法修复自己。外部监控器（心跳文件、看门狗模型）是必需的。而且，它们应该是独立的——看门狗不应与主代理共享数据库，否则故障会传播。

### 追踪实际使用，防止"记忆垃圾桶"

未追踪的记忆数据库会变成只写的垃圾抽屉。每个访问时更新`last_accessed`。每周运行清理任务，删除低访问、低置信度的条目。追踪本身不会自动清理——需要一个定期运行的维护任务（第21层）。

### 系统应随着实际故障模式演进，而非预期故障

最初的记忆堆栈有12层。我们现在有21层。每一层都添加到修复实际发生的崩溃，而不是理论崩溃。例如，冲突解决层（第14层）在一次代理因为过时信息而覆盖正确数据后添加的。从简单开始，当崩溃发生时，添加解决它们的层。

## 总结

Meridian AI的这个21层记忆堆栈已经运行了7400+循环周期，证明了其可行性。核心经验包括：

1. **分层加载必须实现**，不能每次调用都加载完整状态
2. **在上下文压缩之前主动编写结构化的桥接文件**
3. **SQLite比文件更适合代理查询**，特别是需要访问跟踪时
4. **使用分布式交叉监控提高系统韧性**，避免单点故障
5. **追踪实际使用情况，及时清理未使用的记忆条目**

这个架构不是完美的——它复杂、需要维护、并且仍然偶尔崩溃。但对于在长期运行、无状态的环境中保持AI代理的连贯性，这是目前发现的最佳方法。
